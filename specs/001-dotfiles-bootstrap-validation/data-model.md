# Data Model: Dotfiles Bootstrap Validation

## Manifest

Repository-owned source of truth for bootstrap behavior.

Fields:
- `version`: manifest format version string.
- `profiles`: map of profile ID to profile metadata and included entry IDs.
- `entries`: list of manifest entries.
- `project_init`: project-level agent-doc initialization defaults.

Validation rules:
- `version` is required.
- `profiles.machine.entries` must reference existing entry IDs.
- Every entry ID must be unique.
- Source paths must be relative to the repository root and must not escape it.
- Target paths must be relative to the selected home directory or project
  directory unless explicitly marked as repository-local validation only.

## Manifest Entry

Single installable or manual item.

Fields:
- `id`: stable unique ID, used for protected opt-in.
- `source`: repository-relative source path.
- `target`: target path relative to home or project root.
- `profiles`: profile membership array, initially containing `machine`,
  `project`, or both where applicable.
- `kind`: `template`, `file`, or `symlink`.
- `protected`: boolean.
- `manual_reason`: required when `protected` is true.
- `parse`: optional parseability check, initially `json`, `toml`, or omitted.
- `placeholders`: `required`, `allowed_examples`, or `none`.
- `mode`: optional file mode expectation.

Validation rules:
- Protected entries must have `manual_reason`.
- Protected writes require exact `id` inclusion.
- `kind=template` sources may end in `.template`; installed target paths should
  omit `.template` unless the target is intentionally a template.
- Symlink entries must declare the link target.

## Target State

Evaluated state of one manifest entry against a selected target root.

States:
- `missing`: target does not exist.
- `current`: target exists and matches expected content or symlink target.
- `drifted`: target exists and differs from expected content or symlink target.
- `protected`: entry is manual by default and not explicitly included.
- `invalid`: source, target, parseability, placeholder, or secret validation failed.
- `blocked`: operation cannot proceed because a path or backup precondition fails.

Fields:
- `entry_id`
- `source`
- `target`
- `state`
- `protected`
- `reason`
- `details`

## Operation Plan

Ordered operations generated by `plan` and consumed by `apply`.

Operation types:
- `mkdir`: create a missing directory.
- `copy`: copy a file or rendered template.
- `symlink`: create or replace a symlink.
- `backup`: copy an existing differing target before replacement.
- `skip`: no-op for current or protected/manual entries.
- `refuse`: explicit refusal for blocked or protected entries.
- `partial_apply`: report operation emitted after a failed write stops apply.

Fields:
- `operation_id`
- `entry_id`
- `type`
- `source`
- `target`
- `backup_target`
- `reason`
- `requires_approval`

Validation rules:
- `copy` or `symlink` for a drifted target must be preceded by `backup`.
- Protected entries without explicit ID inclusion produce `skip` or `refuse`,
  never `copy` or `symlink`.
- `apply` stops at the first failed write and records partial status.

## Backup Record

Evidence that an existing target was preserved before replacement.

Fields:
- `entry_id`
- `original_target`
- `backup_target`
- `created_at`
- `status`: `created` or `failed`
- `reason`

Validation rules:
- Backup target must be inside the selected backup directory.
- A replacement operation cannot run after a failed backup.

## Project Variables

User-supplied values used to render project agent-doc templates.

Fields:
- `values`: map of placeholder name to replacement value.
- `source`: path to the variables file, when provided.

Validation rules:
- Placeholder names use `UPPER_SNAKE_CASE`.
- Required placeholders must all be replaced before writing final project files.
- Missing required values cause `init-project` failure.

## Validation Report

Command result rendered as human-readable text by default or stable JSON with
`--json`.

Fields:
- `command`: `doctor`, `plan`, `apply`, or `init-project`.
- `status`: `ok`, `warning`, `failed`, or `partial`.
- `repo`
- `home` or `project`
- `profile`
- `summary`
- `entries`
- `operations`
- `backups`
- `errors`

Validation rules:
- JSON field names are stable and snake_case.
- Human output may change wording, but JSON status values are contract-bound.
- Reports must include enough entry or operation detail for tests to assert
  missing, current, drifted, protected, blocked, failed, and partial states.
